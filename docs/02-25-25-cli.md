# Obscura CLI Specification

## Overview

The Obscura CLI supports two execution modes:

1. **Owned Mode (default)**
   Obscura owns the entire request lifecycle:

   * Memory/context injection
   * Tool orchestration
   * Unified streaming events
   * Session management
   * Telemetry

2. **Vendor Passthrough Mode**
   Obscura executes the vendor CLI binary directly and captures transcripts.
   No guarantees are made about structured tool events, usage metadata, or deterministic sessions.

Owned mode is the primary interface. Passthrough mode is an escape hatch.

## Commands

### Owned Chat

Single-turn:

```
obscura chat --backend <provider> "prompt"
```

Interactive:

```
obscura chat --backend claude --session demo --interactive
```

### Agent Execution

```
obscura agent run <agent_id> --backend claude --session demo
```

## Global Flags

* `--backend` openai | localllm | claude | copilot
* `--mode` unified | native (default: unified)
* `--session` explicit session ID
* `--interactive` interactive shell
* `--tools` on | off
* `--tool-policy` auto | none | required:<name>
* `--json` JSON output
* `--no-stream` disable streaming
* `--memory` on | off (default: on)

## Owned Mode Execution Flow

1. Resolve backend
2. Start backend
3. Resolve/create session
4. Load memory/context
5. Build `UnifiedRequest`
6. If `mode=unified`: call `send_request()` or `stream_request()`
7. If `mode=native`: call `backend.native.client` inside pipeline
8. Stream or print output
9. Persist transcript
10. Update memory

Owned mode guarantees structured events and metadata preservation.

## Vendor Passthrough Mode

### Command

```
obscura passthrough <provider> -- <vendor-cli-args>
```

Example:

```
obscura passthrough claude -- chat --model sonnet
```

## Passthrough Guarantees

Passthrough mode:

* Runs vendor CLI binary
* Streams stdout/stderr
* Captures transcript
* Persists transcript in session store

Passthrough mode does NOT guarantee:

* Tool lifecycle capture
* Structured metadata
* Deterministic session continuity
* Unified streaming events

## Design Principles

* Owned mode is deterministic and structured.
* Passthrough mode is convenience-only.
* Unified contract is never polluted with vendor-specific flags.
* Native SDK access is always available via backend adapters.

# Backend Adapter Architecture

## Overview

Obscura supports multiple providers via backend adapters:

* OpenAIBackend
* LocalLLMBackend
* ClaudeBackend
* CopilotBackend

Each backend implements a strict unified interface while exposing raw SDK access.

## Backend Interface

Each backend must implement:

* `start()`
* `stop()`
* `send_request(UnifiedRequest)`
* `stream_request(UnifiedRequest)`
* `create_session()`
* `resume_session()`
* `list_sessions()`
* `delete_session()`
* `register_tool()`
* `register_hook()`
* `capabilities()`
* `native`

## Two Modes

### Unified Mode

* Accepts `UnifiedRequest`
* Emits normalized `StreamChunk` events
* Preserves metadata
* Normalizes tool lifecycle
* Integrates sessions and memory

### Native Mode

* Exposes raw SDK client/session
* No normalization
* No portability guarantees

## Native Handle

Each backend exposes:

* `backend.native.client`
* `backend.native.session` (if supported)
* `backend.native.meta`

Private attributes must not be accessed directly.

## Unified Stream Contract

All backends must emit:

* MESSAGE_START
* TEXT_DELTA
* THINKING_DELTA (optional)
* TOOL_USE_START
* TOOL_USE_DELTA
* TOOL_USE_END
* TOOL_RESULT
* ERROR
* DONE

DONE must include:

* finish_reason
* usage (if available)
* model id
* session id (if available)

Metadata must never be discarded in unified mode.

## Tool Lifecycle

Backend responsibilities:

* Detect tool calls
* Assemble argument deltas
* Emit lifecycle events

Agent runtime responsibilities:

* Execute tool handlers
* Emit TOOL_RESULT
* Inject tool results into conversation

## Sessions

* No reliance on global implicit session
* Session passed explicitly per request
* Conversation history stored per session
* Concurrency-safe behavior required

## Capabilities

Each backend exposes:

BackendCapabilities:

* supports_streaming
* supports_tool_calls
* supports_tool_choice
* supports_usage
* supports_reasoning
* supports_remote_sessions
* supports_multimodal
* supports_mcp

Agent runtime adapts based on capabilities.

## Type Safety

* Use Protocols for SDK clients
* `_ensure_client()` returns typed client
* Provide `.pyi` stubs for untyped SDKs
* Unified layer must be Pyright-safe
* Avoid leaking `Any` across unified boundary

# CLI Code Structure

## Directory Layout

src/obscura/
cli/
**init**.py

passthrough.py

base.py

copilot_backend.py
runtime/
agent_loop.py
memory_engine.py
session_store.py
types/
unified_request.py
stream_chunk.py
message.py

## CLI Entry Point

main.py

Responsibilities:

* Parse CLI arguments
* Route to subcommands
* Initialize backend factory

## chat.py

Responsibilities:

* Owned chat mode
* Interactive loop
* Memory injection
* Unified vs native routing

Execution:

```
if mode == "unified":
    call backend.stream_request(req)
else:
    call backend.native.client...
```

## agent.py

Responsibilities:

* Multi-turn agent execution
* Tool orchestration
* Session persistence
* Streaming unified events

Delegates to AgentLoop.

## passthrough.py

Responsibilities:

* Validate vendor CLI exists
* Spawn subprocess
* Stream stdout/stderr
* Capture transcript
* Persist transcript

Must not depend on backend adapter logic.

## utils.py

Contains:

* Backend factory
* Session resolution helpers
* Memory injection utilities
* JSON formatter
* Streaming printer

## Testing Strategy

Owned CLI:

* Snapshot JSON output
* Verify DONE includes metadata
* Verify session transcript stored
* Verify tool lifecycle ordering

Passthrough CLI:

* Mock subprocess invocation
* Validate command construction
* Validate transcript capture
* Avoid real vendor CLI in CI

## Design Constraints

* Unified contract remains strict
* Passthrough is clearly separated
* No provider-specific flags leak into unified mode
* Session state must be concurrency-safe
* CLI output must be testable via JSON mode

# Obscura Architecture

## Overview

Obscura provides a unified agent runtime that supports multiple model providers while preserving full native SDK access when needed.

It operates in **two explicit modes**:

1. **Unified Mode** — Stable, normalized interface for multi-agent orchestration, tools, sessions, hooks, streaming, telemetry, and memory.
2. **Native Mode** — Direct access to provider SDKs with zero abstraction constraints.

This architecture allows:

* Clean multi-provider support (OpenAI, Local OpenAI-compatible, Claude, Copilot)
* Strict, testable unified behavior
* Full vendor feature access without polluting the unified contract

---

# Core Design Principles

1. **Two Modes, Never Mixed**

   * Unified mode does not accept provider-specific kwargs.
   * Native mode does not promise cross-provider portability.

2. **Adapters, Not Wrappers**
   Each backend translates between:

   * Unified types → Provider request
   * Provider events → Unified stream events
   * Provider responses → Unified message objects

3. **Unified Contract is Stable**
   Multi-agent runtime depends only on unified types.
   Providers may evolve independently.

4. **Capabilities Over Pretending**
   Backends declare what they support.
   Agent runtime adapts accordingly.

# System Layers

## Layer 0 — Provider SDKs

* OpenAI SDK
* OpenAI-compatible local servers
* Claude Agent SDK
* GitHub Copilot SDK

These are treated as external engines.

## Layer 1 — Backend Adapters

Each backend implements:

* `start()`
* `stop()`
* `send_request(UnifiedRequest)`
* `stream_request(UnifiedRequest)`
* `create_session()`
* `resume_session()`
* `list_sessions()`
* `delete_session()`
* `register_tool()`
* `register_hook()`
* `capabilities()`
* `native` (raw access)

### Backends

* `OpenAIBackend`
* `LocalLLMBackend`
* `ClaudeBackend`
* `CopilotBackend`

Each backend is responsible for:

* Request translation
* Stream normalization
* Tool lifecycle normalization
* Metadata preservation
* Session handling

---

## Layer 2 — Agent Runtime

Shared across all backends.

Responsibilities:

* Multi-agent orchestration
* Tool execution
* Tool result injection
* Memory/context engine
* Hook execution
* Telemetry
* Retry / timeout logic

Agent runtime depends only on unified types.

# Two Operating Modes

## Unified Mode

Used for:

* Multi-agent systems
* Tool execution
* Memory integration
* Session management
* Streaming control
* Cross-provider portability

### Unified Entry Points

* `send_request(req: UnifiedRequest) -> Message`
* `stream_request(req: UnifiedRequest) -> AsyncIterator[StreamChunk]`

### UnifiedRequest

```
UnifiedRequest:
    messages: list[ChatMessage]
    params: CompletionParams
    tool_policy: ToolPolicy
    session: SessionRef | None
    metadata: dict[str, Any] | None
    timeout_s: float | None
    cancel_token: CancelToken | None
```

No provider-specific kwargs are accepted in unified mode.

---

## Native Mode

Used for:

* Provider-specific features
* Experimental features
* CLI integrations
* Advanced vendor routing
* Non-portable behavior

### Native Access

```
backend.native.client
backend.native.session
backend.native.meta
```

Native mode provides full SDK control.

Unified abstractions are bypassed.

---

# Unified Stream Contract

All backends must emit the same normalized stream events:

* `MESSAGE_START`
* `TEXT_DELTA`
* `THINKING_DELTA` (optional)
* `TOOL_USE_START`
* `TOOL_USE_DELTA`
* `TOOL_USE_END`
* `TOOL_RESULT`
* `ERROR`
* `DONE`

## DONE Event Requirements

`DONE` must include:

* finish_reason
* usage (if available)
* model id
* session_id (if available)

Metadata must never be discarded in unified mode.

---

# Tool Lifecycle

## Backend Responsibility

* Detect tool calls
* Assemble tool argument deltas
* Emit:

  * TOOL_USE_START
  * TOOL_USE_DELTA
  * TOOL_USE_END

## Agent Runtime Responsibility

* Execute tool handler
* Emit TOOL_RESULT
* Inject tool result into conversation
* Continue model execution if required

Tool execution logic is not vendor-specific.

---

# Sessions

Unified sessions are explicitly managed.

Rules:

* No reliance on global active session
* Session passed per request when needed
* Backends may map unified sessions to native sessions when supported
* Conversation history stored per session

Concurrency must not rely on implicit global state.

---

# Capabilities

Each backend exposes:

```
BackendCapabilities:
    supports_streaming: bool
    supports_tool_calls: bool
    supports_tool_choice: bool
    supports_usage: bool
    supports_reasoning: bool
    supports_remote_sessions: bool
    supports_multimodal: bool
    supports_mcp: bool
```

Agent runtime adapts based on capabilities.

Unified mode never silently ignores unsupported features.

---

# CLI Strategy

Vendor CLI processes are not embedded directly.

Instead:

* Provide `obscura` CLI
* Internally use appropriate backend
* Integrate memory/context engine
* Maintain unified tool lifecycle

Optional future feature:

* Native CLI passthrough mode (non-integrated vendor CLI bridge)

---

# Packaging Strategy

Initial approach:

```
pip install obscura
```

Installs all provider SDKs.

Future option:

```
pip install obscura[openai]
pip install obscura[claude]
pip install obscura[copilot]
pip install obscura[all]
```

Core runtime remains provider-agnostic.

---

# Type Safety Strategy

* Define Protocols for SDK clients
* `_ensure_client()` returns typed protocol
* Provide local `.pyi` stubs for untyped SDKs
* Unified types fully typed and Pyright-safe
* No `Any` leaks across unified boundary

---

# Error Handling

Unified mode guarantees:

* Normalized ERROR events
* DONE always emitted (unless cancellation)
* Hook failures do not crash runtime
* Timeouts and cancellation supported

Native mode behavior is provider-defined.

# Summary

Obscura provides:

* Clean backend adapters
* Strict unified contract
* Full native SDK access
* Multi-agent orchestration
* Memory/context integration
* Cross-provider portability

The architecture intentionally separates:

* Provider-specific behavior
* Unified agent runtime behavior

This prevents abstraction drift, metadata loss, and feature suppression while enabling advanced multi-agent systems across multiple model providers.
